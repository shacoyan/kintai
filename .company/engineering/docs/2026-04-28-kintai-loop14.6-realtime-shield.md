# L14.6 緊急バグ修正 — Realtime Shield 横展開（3 Hooks）

## 概要
- **何を**: `useActiveAttendance` / `useCorrection` / `useAttendance` の Supabase Realtime `channel().on().subscribe()` を、L14.5 で `useNotification` に適用した Shield パターンと同じ防御で包む。
- **なぜ**: 各ページ（打刻 / 履歴 / シフト / ダッシュボード）で `予期しないエラーが発生しました`（ErrorBoundary 発火）が再現中。`useNotification` を Shield 化（commit 8e574ac）しても解消しないことから、同パターンの未防御 subscribe が他 hook に残存していると断定。WebSocket setup / subscribe コールバックでスローされた例外が React レンダーツリーに伝播し、ErrorBoundary を発火させているのが直接原因。
- **ゴール**: 3 hook の Realtime 経路を完全に「投げない」状態にし、失敗時は `console.warn` でフォールバック（ポーリング/手動 fetch）に委ねる。UI は壊さない。

## 参考実装（L14.5 / commit 8e574ac）
`src/hooks/useNotification.ts` L92–L148 が完成形。以下の 4 防御層を持つ:

1. **userId/tenantId Gate**: 未確定なら何もせず return（既存）
2. **冪等 cleanup**: subscribe 前に `channelRef.current` を `try/catch + removeChannel` で破棄
3. **try/catch で channel 構築をラップ**:
   - `let channel: ReturnType<typeof supabase.channel> | null = null;` を先に宣言
   - `channel = supabase.channel(...).on(...).subscribe((status, err) => { if (CHANNEL_ERROR/TIMED_OUT/err) console.warn(...) })`
   - `channelRef.current = channel`
   - catch: `console.warn` + `setError(...)` + 失敗した channel があれば `removeChannel` + `channelRef.current = null` + `return`
4. **return cleanup を try/catch で removeChannel 冪等化**

> 重要: `.subscribe()` のステータスコールバックを必ず付与し、`CHANNEL_ERROR` / `TIMED_OUT` / `err` を warn ログに落とす。ここを省くと WebSocket 失敗が黙って再接続ループに入り、別経路でエラーを生む。

3. L297–L317 の useEffect を以下構造へ Shield 化:
   ```
   useEffect(() => {
     if (!tenantId) return;

     // 冪等 cleanup
     if (channelRef.current) {
       try { supabase.removeChannel(channelRef.current); } catch (e) { console.warn('[useCorrection] removeChannel failed:', e); }
       channelRef.current = null;
     }

     let channel: ReturnType<typeof supabase.channel> | null = null;
     try {
       channel = supabase
         .channel(`correction_requests:${tenantId}`)
         .on(
           'postgres_changes',
           {
             event: '*',
             schema: 'public',
             table: 'correction_requests',
             filter: `tenant_id=eq.${tenantId}`,
           },
           () => { void fetchRequests(); }
         )
         .subscribe((status, err) => {
           if (status === 'CHANNEL_ERROR' || status === 'TIMED_OUT' || err) {
             console.warn('[useCorrection] subscribe status:', status, err);
           }
         });
       channelRef.current = channel;
     } catch (e) {
       console.warn('[useCorrection] channel setup failed:', e);
       setError(e instanceof Error ? e.message : String(e));
       if (channel) {
         try { supabase.removeChannel(channel); } catch (re) { console.warn('[useCorrection] removeChannel after fail:', re); }
       }
       channelRef.current = null;
       return;
     }

     return () => {
       if (channelRef.current) {
         try { supabase.removeChannel(channelRef.current); } catch (e) { console.warn('[useCorrection] removeChannel failed:', e); }
         channelRef.current = null;
       }
     };
   }, [tenantId, fetchRequests]);
   ```

## 呼び出し元コンポーネントの Gate パターン分析

L14.5 の `NotificationBell` は `useAuth().user` 未確定時に `return null` する Gate を導入（commit 8e574ac）。今回の3 hook について同様の Gate が必要か検証した結果:

| 呼び出し元 | tenantId | storeId | userId | Gate 要否 |
|---|---|---|---|---|
| `DashboardPage` (`useAttendance`) | `currentTenant!.id` | `currentStore?.id ?? null` | hook 内で `auth.getUser()` | **不要**（hook 内で storeId null 時 early return 済み） |
| `ActiveMembersCard` (`useActiveAttendance`) | props (TenantContext 経由) | props (`null` 許容) | — | **不要**（hook 内 early return 済み） |
| `HistoryPage` (`useCorrection`) | `currentTenant!.id` | — | — | **不要**（hook 内に `if (!tenantId) return;` を新設するため） |
| `AdminDashboard` (`useCorrection`) | `currentTenant!.id` | — | — | **不要**（同上） |
| `CorrectionForm` (`useCorrection`) | props | — | — | **不要**（同上） |

**結論**: いずれの呼び出し元も `RequireTenant` ガードを通過済みで `currentTenant!` を保証しており、Bell 同等の Gate は不要。Shield 内部に既存の null/undefined Guard があるため、hook 内防御で十分。

ただし将来 `currentTenant` が null になり得る経路が増えた場合（マルチテナント切替中など）は、各呼び出し元で `if (!currentTenant) return null;` 相当のガードを足すこと。今回のスコープ外。

## 統合時の注意点

1. **既存 `channelRef` / `cleanupChannel` を消さない** — L14.5 の useNotification は ref ベースだったが本タスクでは2 hook に既に存在するため、二重定義しないよう留意。
2. **subscribe ステータスコールバックを必ず付与** — これを忘れると WebSocket 失敗が無音化して別経路で再現する。
3. **fetch callback (例: `() => fetchActive()`) は try/catch で包まない** — ここで投げてもイベントループ上で処理されるだけで、hook の Realtime 接続そのものは死なない。過剰防御は可読性を下げる。
4. **ポーリング useEffect は変更禁止** — Shield と分離されており、Realtime が死んでも 60秒ポーリングで劣化動作する設計。
5. **`useCorrection` の `useRef` import 追加忘れ** に注意。型エラーが即出る。
6. **テスト**: ローカルで `npm run dev`、以下4ページの E2E 簡易確認:
   - `/` (DashboardPage) — 打刻ボタン表示
   - `/history` (HistoryPage) — 修正申請一覧表示
   - `/admin` (AdminDashboard) — 出勤中メンバー / 修正申請審査
   - 打刻ページ — 打刻可能
   いずれも `予期しないエラーが発生しました` 画面が出ないこと。Console に `[hookName] subscribe status:` が warn として出ても OK（むしろ正しい挙動）。
7. **ビルド**: `npm run build` がエラーゼロで通ること。
8. **commit message 規約**: `fix(kintai): L14.6 Realtime subscribe を Shield 化（useActiveAttendance / useCorrection / useAttendance）`
9. **dual push**: newWorld + kintai/main へ subtree push。
